# Logos CLI — Design Spec

## Overview

Logos is an open-source Node.js CLI tool that scaffolds personal AI research projects. It creates a structured directory with a staged research workflow, powered by Orchestra Research's AI Research Skills running inside Claude Code.

The user installs `logos` via npm, runs `logos <project-name>`, and gets a research-ready workspace. The actual research is conducted by Claude Code inside the project — `logos` itself is a scaffolder, not an orchestrator.

## CLI Interface

```
logos <project-name>    # Scaffold a new research project
logos --status          # Show current stage (run from inside a project)
logos --update          # Update Orchestra Research skills (run from inside a project)
```

Published to npm. The package registers a `logos` binary. If the name `logos` is taken, fallback to `logos-research` or `@logos-research/cli`.

Standard flags: `logos --help`, `logos --version`.

### Error Handling

- If `git` is not installed, exit with a clear error message.
- If the target directory already exists, exit with an error (do not overwrite).
- If `--status` or `--update` is run outside a logos project (no `.logos` marker file found), exit with an error.
- If the submodule clone fails (network error), scaffold the project anyway and print instructions to retry with `logos --update`.

## Scaffolded Project Structure

```
<project-name>/
├── .claude/
│   └── skills/              # Orchestra Research skills (git submodule)
├── .logos                       # Marker file (identifies this as a logos project)
├── .env.example             # Template showing supported API keys
├── .gitignore               # Ignores .env, large files, data/
├── CLAUDE.md                # Workflow instructions for Claude Code
├── README.md                # Project overview
├── agenda.md                # Research agenda (populated during onboarding)
├── setup.md                 # Research infrastructure & preferences (populated during onboarding)
├── stages/                  # Empty — stage files created as stages complete
├── papers/                  # PDFs, references, literature
├── experiments/             # Code, notebooks, results
├── notes/                   # Freeform research notes
└── data/                    # Datasets, raw data
```

Git is initialized with an initial commit. Orchestra Research skills are added as a git submodule at `.claude/skills/`.

## Onboarding Flow

When the user first opens the project in Claude Code, `CLAUDE.md` instructs the agent to check if `agenda.md` and `setup.md` are still in their template state (headings with HTML comments only). If so, the agent runs an interactive onboarding:

1. Ask the user about their research goals, questions, and scope → populate `agenda.md`
2. Ask about compute, storage, tracking, and tooling preferences → populate `setup.md`

Only after both files are populated does the agent proceed to the staged workflow.

## Staged Workflow

### Stages

1. **Research** — Literature review + synthesis. Sub-agents run in parallel to search, retrieve, and summarize papers. The agent synthesizes findings, identifies gaps, and forms hypotheses.
2. **Experimentation** — Write code, run experiments, collect results using the infrastructure described in `setup.md`.
3. **Analysis** — Interpret results, draw conclusions, identify follow-up questions.
4. **Writing** — Produce a writeup or paper.

### Stage Mechanics

- **Stage files as state:** The `stages/` directory is the source of truth. A stage is complete when its file exists (e.g. `stages/1-research.md`). No file = not started or in progress.
- **Stage file content:** Auto-generated by the agent on completion. Contains a summary of what was done, key findings, and artifacts produced.
- **Gate rule:** The agent must never begin a stage unless the previous stage's file exists. It must always ask the user for explicit permission before starting the next stage.
- **Resumability:** On every session start, the agent reads `agenda.md`, `setup.md`, and scans `stages/` to determine current progress. If a stage file exists, its contents are treated as ground truth — the agent builds on it rather than redoing work.
- **Backpedaling:** The user can delete a stage file to redo that stage. The agent detects the absence and picks up from there.
- **Cost awareness:** The agent warns the user before any action that incurs compute or API costs.

### Sub-Agent Usage in Research Stage

`CLAUDE.md` explicitly instructs the agent to use sub-agents during the Research stage for parallel work:
- Dispatch sub-agents to search for and summarize papers concurrently
- A coordinating agent synthesizes the sub-agent outputs into a coherent literature review
- This enables broader coverage without sequential bottleneck

## CLAUDE.md Rules

The `CLAUDE.md` file encodes these instructions for Claude Code:

1. **Session start:** Read `agenda.md` for research goals, `setup.md` for infrastructure, `.env` for API keys, and scan `stages/` for current progress.
2. **Onboarding gate:** If `agenda.md` or `setup.md` are in template state, run onboarding before anything else.
3. **Stage order:** Research → Experimentation → Analysis → Writing.
4. **Gate rule:** Never begin a stage without the previous stage file existing. Always ask for explicit user permission.
5. **Stage completion:** Auto-generate the stage file with summary, findings, and artifacts.
6. **Resumability:** Always check `stages/` at session start. Build on completed stage files, don't redo.
7. **Cost awareness:** Warn before compute/API-cost actions.
8. **Sub-agents:** Use parallel sub-agents in the Research stage for literature search and summarization.
9. **Skills:** Use Orchestra Research skills from `.claude/skills/` as needed for the current stage.

## API Keys & Environment

- `.env` at project root holds API keys (`WANDB_API_KEY`, `RUNPOD_API_KEY`, `HF_TOKEN`, etc.)
- `.env.example` ships with the scaffold listing supported keys (no values)
- `.env` is gitignored
- `CLAUDE.md` instructs the agent to read `.env` at session start if it exists. If `.env` is absent, the agent should not error — it proceeds and only prompts for keys when a stage action requires them.

## Skills Installation & Updates

- Orchestra Research skills are installed as a git submodule at `.claude/skills/` during `logos <project-name>`
- Source: `https://github.com/Orchestra-Research/AI-Research-SKILLs`
- Update via `logos --update` (runs `git submodule init` if needed, then `git submodule update --remote`)
- When cloning a logos project, users must run `git submodule init && git submodule update` — the scaffolded README will document this
- Skills are loaded on-demand by the agent per CLAUDE.md — the agent reads only the skills relevant to the current stage, not all 89 at once, to conserve context window

## .gitignore

```
.env
node_modules/
```

Minimal by default — only secrets and tool artifacts are ignored. Large files (weights, datasets) are left to the user to manage via Git LFS or their own `.gitignore` additions. This avoids silent data loss.

## Tech Stack

- **Runtime:** Node.js
- **Package:** Published to npm
- **Dependencies:** Minimal — likely just a CLI argument parser (e.g. `commander` or `yargs`) and `child_process` for git commands
- **No AI SDK dependency** — `logos` does not call any AI APIs. It scaffolds; Claude Code does the research.
- **Minimum Node.js:** 18+

## Template File Contents

### agenda.md (template state)

```markdown
# Research Agenda

## Overview
<!-- Describe your research topic and motivation -->

## Questions
<!-- What questions are you trying to answer? -->

## Goals
<!-- What does success look like? -->

## References
<!-- Any starting references or prior work -->
```

### setup.md (template state)

```markdown
# Research Setup

## Compute
<!-- e.g. Lambda Labs for training, Modal for inference, local RTX 4090 -->

## Storage
<!-- e.g. HuggingFace Hub for weights, S3 for datasets -->

## Tracking
<!-- e.g. Weights & Biases for experiment tracking -->

## Other Tools
<!-- Any other services or preferences -->
```

### README.md

Two READMEs exist in this project:

1. **Repo README** (`logos/README.md`) — the open-source project page on GitHub. This is the landing page. Must be polished, compelling, and visually well-crafted. Clearly communicate what logos is, why it exists, how to install and use it. Should make people want to try it.
2. **Scaffolded project README** (`<project-name>/README.md`) — a living document inside each research project. Structured to present: current research stage, key findings so far, project setup, and directory layout. The agent updates this as stages complete, so it serves as an at-a-glance view of the research.

### Template State Detection

The onboarding gate detects template state via a `<!-- TEMPLATE -->` sentinel comment at the top of each file. The scaffolder includes this sentinel; the onboarding flow removes it after populating the file. This is more robust than content heuristics.

## Stage-to-Skill Mapping

The agent selects skills on demand, but CLAUDE.md provides guidance:

| Stage | Recommended Skills |
|-------|-------------------|
| Research | Autoresearch, Ideation, ML Paper Writing (for reading), RAG tools |
| Experimentation | Domain-specific skills based on `setup.md` (e.g. Fine-Tuning, Distributed Training, Infrastructure) |
| Analysis | Evaluation, MLOps/tracking, Mechanistic Interpretability |
| Writing | ML Paper Writing |

This mapping is advisory — the agent uses its judgment to load additional skills as the research demands.
